00024 // KissXML has rather straight-forward memory management:
00025 // https://github.com/robbiehanson/KissXML/wiki/MemoryManagementThreadSafety
00026 // 
00027 // There are 3 important concepts to keep in mind when working with KissXML:
00028 // 
00029 // 
00030 // 1.) KissXML provides a light-weight wrapper around libxml.
00031 // 
00032 // The parsing, creation, storage, etc of the xml tree is all done via libxml.
00033 // This is a fast low-level C library that's been around for ages, and comes pre-installed on Mac OS X and iOS.
00034 // KissXML provides an easy-to-use Objective-C library atop libxml.
00035 // So a DDXMLNode, DDXMLElement, or DDXMLDocument are simply objective-c objects
00036 // with pointers to the underlying libxml C structure.
00037 // Then only time you need to be aware of any of this is when it comes to equality.
00038 // In order to maximize speed and provide read-access thread-safety,
00039 // the library may create multiple DDXML wrapper objects that point to the same underlying xml node.
00040 // So don't assume you can test for equality with "==".
00041 // Instead use the isEqual method (as you should generally do with objects anyway).
00042 // 
00043 // 
00044 // 2.) XML is implicitly a tree heirarchy, and the XML API's are designed to allow traversal up & down the tree.
00045 // 
00046 // The tree heirarchy and API contract have an implicit impact concerning memory management.
00047 // 
00048 // <starbucks>
00049 //   <latte/>
00050 // </starbucks>
00051 // 
00052 // Imagine you have a DDXMLNode corresponding to the starbucks node,
00053 // and you have a DDXMLNode corresponding to the latte node.
00054 // Now imagine you release the starbucks node, but you retain a reference to the latte node.
00055 // What happens?
00056 // Well the latte node is a part of the xml tree heirarchy.
00057 // So if the latte node is still around, the xml tree heirarchy must stick around as well.
00058 // So even though the DDXMLNode corresponding to the starbucks node may get deallocated,
00059 // the underlying xml tree structure won't be freed until the latte node gets dealloacated.
00060 // 
00061 // In general, this means that KissXML remains thread-safe when reading and processing a tree.
00062 // If you traverse a tree and fork off asynchronous tasks to process subnodes,
00063 // the tree will remain properly in place until all your asynchronous tasks have completed.
00064 // In other words, it just works.
00065 // 
00066 // However, if you parse a huge document into memory, and retain a single node from the giant xml tree...
00067 // Well you should see the problem this creates.
00068 // Instead, in this situation, copy or detach the node if you want to keep it around.
00069 // Or just extract the info you need from it.
00070 // 
00071 // 
00072 // 3.) KissXML is read-access thread-safe, but write-access thread-unsafe (designed for speed).
00073 // 
00074 // <starbucks>
00075 //   <latte/>
00076 // </starbucks>
00077 // 
00078 // Imagine you have a DDXMLNode corresponding to the starbucks node,
00079 // and you have a DDXMLNode corresponding to the latte node.
00080 // What happens if you invoke [starbucks removeChildAtIndex:0]?
00081 // Well the undelying xml tree will remove the latte node, and release the associated memory.
00082 // And what if you still have a reference to the DDXMLNode that corresponds to the latte node?
00083 // Well the short answer is that you shouldn't use it. At all.
00084 // This is pretty obvious when you think about it from the context of this simple example.
00085 // But in the real world, you might have multiple threads running in parallel,
00086 // and you might accidently modify a node while another thread is processing it.
00087 // 
00088 // To completely fix this problem, and provide write-access thread-safety, would require extensive overhead.
00089 // This overhead is completely unwanted in the majority of cases.
00090 // Most XML usage patterns are heavily read-only.
00091 // And in the case of xml creation or modification, it is generally done on the same thread.
00092 // Thus the KissXML library is write-access thread-unsafe, but provides speedier performance.
00093 // 
00094 // However, when such a bug does creep up, it produces horrible side-effects.
00095 // Essentially the pointer to the underlying xml structure becomes a dangling pointer,
00096 // which means that accessing the dangling pointer might give you the correct results, or completely random results.
00097 // And attempting to make modifications to non-existant xml nodes via the dangling pointer might do nothing,
00098 // or completely corrupt your heap and cause un-explainable crashes in random parts of your library.
00099 // Heap corruption is one of the worst problems to track down.
00100 // So to help out, the library provides a debugging macro to track down these problems.
00101 // That is, if you invalidate the write-access thread-unsafe rule,
00102 // this macro will tell you when you're trying to access a now-dangling pointer.
00103 // 
00104 // How does it work?
00105 // Well everytime a DDXML wrapper object is created atop a libxml structure,
00106 // it marks the linkage in a table.
00107 // And everytime a libxml structure is freed, it destorys all corresponding linkages in the table.
00108 // So everytime a DDXML wrapper objects is about to dereference it's pointer,
00109 // it first ensures the linkage still exists in the table.
00110 // 
00111 // Set to 1 to enable
00112 // Set to 0 to disable (this is the default)
00113 // 
00114 // The debugging macro adds a significant amount of overhead, and should NOT be enabled on production builds.
00115
